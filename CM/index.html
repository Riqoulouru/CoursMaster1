Partie 1

Pourquoi une To-Do List a besoin d’une API

  • L’utilisateur veut voir ses tâches sur mobile, web ou desktop.
  • Les données doivent rester au même endroit.
  • Plusieurs clients doivent pouvoir parler au même système.
  • L’API sert d’intermédiaire entre l’interface et les données.

API

Définition simple d’une API

  • Une API est une porte d’entrée vers une application.
  • Elle reçoit des demandes et renvoie des réponses.
  • Elle expose des règles claires sur ce qu’on peut faire.
  • Dans notre cas : gérer des tâches.
Système

Client, serveur, base de données

  • Le client envoie une demande.
  • Le serveur contient l’API.
  • L’API parle à la base de données.
  • La réponse repart vers le client.
HTTP

HTTP sert à transporter les demandes

  • Le navigateur et l’application mobile parlent souvent en HTTP.
  • HTTP transporte une méthode, une URL et parfois un corps.
  • Le serveur renvoie un statut et des données.
  • Exemple : demander la liste des tâches.
GET /tasks HTTP/1.1 Host: todo.local HTTP/1.1 200 OK Content-Type: application/json
Analogie HTTP ressemble à une enveloppe qui transporte une demande compréhensible par les deux côtés.

  • Spring Boot simplifie le démarrage du projet.
  • Le développeur se concentre davantage sur le besoin métier.
  • Spring est extremement utilise en entreprise, donc l'apprendre est un choix tres rentable.

Framework

Un framework donne une structure prête à l’emploi

  • Il propose des conventions.
  • Il évite de réinventer des briques communes.
  • Il aide à brancher facilement web, base et sécurité.
  • Il standardise la façon de coder en équipe.
Analogie Un framework ressemble à un chantier avec déjà les fondations, les outils et le plan général.
Structure

Controller, Service, Repository : trois rôles simples

  • Le Controller reçoit la requête HTTP.
  • Le Service porte la logique métier.
  • Le Repository gère l’accès aux données.
  • Cette séparation rend le code plus lisible.
Controller

Le controller parle HTTP

  • Il écoute une URL.
  • Il lit les paramètres, le corps et l’identifiant.
  • Il appelle le service approprié.
  • Il renvoie une réponse HTTP.
@RestController @RequestMapping("/tasks") class TaskController { @GetMapping List all() { ... } }
Question Pourquoi le controller ne devrait-il pas contenir toute la logique métier ?
Service

Le service porte les règles métier

  • Il décide ce qu’il faut faire.
  • Il vérifie les conditions métier.
  • Il orchestre les appels nécessaires.
  • Dans la To-Do List : refuser un titre vide, par exemple.
@Service class TaskService { private final TaskRepository repository; List findAll() { return repository.findAll(); } Task create(Task task) { return repository.save(task); } }
Repository

Le repository parle à la base de données

  • Il sauvegarde les tâches.
  • Il relit les tâches existantes.
  • Il masque les détails techniques d’accès aux données.
  • Avec Spring Data, beaucoup de code est évité.
@Repository interface TaskRepository extends JpaRepository { }
@Entityobjet stocké
@Ididentifiant unique

Injection

Injection de dépendances et IoC

  • Une classe utilise d’autres classes pour travailler.
  • Spring crée ces objets et les relie.
  • C’est l’idée d’Inversion of Control.
  • On préfère souvent l’injection par constructeur.
class TaskController { private final TaskService service; TaskController(TaskService service) { this.service = service; } }
Explication simple Au lieu de fabriquer ses dépendances seul, le controller les reçoit déjà prêtes.
Injection

IoC : qui construit les objets ?

  • Sans framework, une classe peut créer elle-même ses dépendances.
  • Avec Spring, le conteneur crée et relie les objets.
  • C’est cela qu’on appelle l’Inversion of Control.
  • Le développeur décrit les briques, Spring gère l’assemblage.
// Sans IoC class TaskController { private final TaskService service = new TaskService(); } // Avec IoC / Spring class TaskController { private final TaskService service; TaskController(TaskService service) { this.service = service; } }

  • Le service applique les règles.

Le cycle complet d’une requête : le retour

  • La base renvoie les données sauvegardées.
  • Le repository renvoie le résultat au service.
  • Le service prépare la réponse utile.
  • Le controller renvoie une réponse HTTP en JSON.

Monolithe

Monolithe : une seule application principale

  • Le code de l’API est dans un seul projet.
  • Le déploiement se fait souvent en un seul bloc.
  • C’est simple à démarrer.
  • C’est souvent un bon premier choix pour une To-Do List.
Microservices

Microservices : plusieurs petits services spécialisés

  • Chaque service porte une responsabilité plus ciblée.
  • Les services communiquent entre eux.
  • Cela apporte plus de souplesse, mais plus de complexité.
  • Pour une petite To-Do List, c’est souvent trop tôt.
Task servicegérer les tâches
User servicegérer les comptes
Notification servicerappels et emails
Microservices

Comment fonctionne une architecture microservices

  • Chaque service possède une responsabilité métier bien délimitée.
  • Chaque service expose souvent sa propre API.
  • Les services se parlent via HTTP, messages ou événements.
  • Chaque service peut être déployé séparément.
Microservices

Comment construire une architecture microservices

  • Commencer par découper le métier en domaines simples.
  • Donner à chaque service une responsabilité claire.
  • Définir les échanges entre services : API, événements, messages.
  • Prévoir le déploiement, la supervision et la maintenance.
  • Éviter de découper trop tôt ou trop finement.
1. Découpertâches, utilisateurs, notifications
2. Contractualiserroutes et messages clairs
3. Isolerchaque service et son cycle de vie
4. Surveillerlogs, erreurs, disponibilité
Réflexe utile On ne construit pas des microservices pour faire moderne, mais pour répondre à un vrai besoin d’organisation ou de scalabilité.
Comparaison

Monolithe vs microservices

  • Le monolithe est plus simple à lancer et à déployer.
  • Les microservices isolent mieux certaines responsabilités.
  • Les microservices coûtent plus cher en coordination.
  • Le contexte réel doit guider le choix.

Classique

L’architecture Spring classique est simple à comprendre

  • On sépare souvent par couches : controller, service, repository.
  • C’est cohérent avec beaucoup de tutoriels Spring.
  • Elle fonctionne bien sur des projets petits à moyens.
  • Elle est facile à lire pour débuter.
todo |-- controller |-- service |-- repository `-- entity
Limites

La version classique a aussi des limites

  • La logique métier peut glisser dans le controller ou le repository.
  • Les dépendances techniques peuvent envahir le cœur du code.
  • Les tests métier deviennent parfois plus lourds.
  • Le changement d’outil technique peut être coûteux.
Symptôme fréquent un controller qui valide tout, construit tout et parle déjà à la base
Hexagonal

L’idée de l’architecture hexagonale

  • Le métier est placé au centre.
  • La technique reste autour du métier.
  • Les interactions passent par des ports.
  • Les éléments techniques deviennent des adaptateurs.
Ressource complémentaire Voir la vidéo sur l’architecture hexagonale Voir la vidéo sur l’architecture hexagonale
Ports

Ports et adaptateurs, version simple

  • Un port décrit un besoin ou une capacité.
  • Un adaptateur entrant déclenche l’action, par exemple le REST.
  • Un adaptateur sortant réalise un besoin technique, par exemple JPA.
  • Le métier dépend d’interfaces, pas d’outils concrets.
public interface CreateTaskUseCase { Task handle(CreateTaskCommand command); } public interface SaveTaskPort { Task save(Task task); }
Hexagonal

Comment lire le schéma hexagonal

  • Le REST est un point d’entrée.
  • Le métier reste au centre de la décision.
  • La base ou le mail sont des sorties techniques.
  • Le cœur métier reste indépendant de ces outils.

Erreurs

Erreurs classiques à éviter

  • Mettre toute la logique dans le controller.
  • Mélanger règles métier et détails SQL partout.
  • Choisir microservices ou hexagonal “par mode”.
  • Écrire trop de couches pour un besoin trop simple.
  • Oublier de nommer clairement les responsabilités.
Réflexe utile avant d’ajouter de l’architecture, demander : quel problème concret est-ce que je résous ?
API Réelle

Sécurité et erreurs : une API ne renvoie pas juste “ça marche”

  • Une API doit souvent vérifier qui appelle et ce qu’il a le droit de faire.
  • On parle alors d’authentification, d’autorisation et parfois de rôles.
  • Les entrées doivent aussi être validées avant de traiter la requête.
  • Les erreurs doivent être claires : `400`, `401`, `403`, `404`, `500`.
  • Une réponse d’erreur cohérente aide autant les développeurs que les clients.
HTTP/1.1 404 Not Found Content-Type: application/json { "error": "TASK_NOT_FOUND", "message": "La tache 42 n'existe pas" }

Authentification, autorisation, rôles : trois idées différentes

  • Authentification : qui es-tu ?
  • Autorisation : as-tu le droit de faire cette action ?
  • Rôles : admin, utilisateur simple, manager, etc.
  • Une API réelle vérifie souvent ces trois dimensions en même temps.

API Réelle

Une bonne erreur d’API doit aider à comprendre vite

  • Le code HTTP donne une catégorie d’erreur.
  • Le corps de réponse donne un message exploitable.
  • Le client doit comprendre s’il doit corriger sa requête ou réessayer plus tard.
  • Une erreur floue ralentit le debug et la maintenance.
{ "error": "VALIDATION_ERROR", "message": "Le titre est obligatoire", "field": "title" }
Réflexe utile une API ne doit pas seulement échouer, elle doit échouer proprement.
API Réelle

Documentation et versioning rendent l’API maintenable

  • Une API utile doit être facile à comprendre pour ses consommateurs.
  • Swagger / OpenAPI servent à décrire les routes, paramètres, formats et erreurs.
  • La documentation sert aussi de contrat entre backend et frontend.
  • Quand l’API évolue, il faut éviter de casser les anciens clients.
  • Le versioning aide à faire évoluer l’API plus proprement.

Le versioning sert à faire évoluer sans casser

  • Un client mobile ou frontend peut dépendre d’un format précis.
  • Changer brutalement l’API peut casser ces clients.
  • Versionner permet d’introduire une nouvelle version progressivement.
  • Il faut donc penser compatibilité dès qu’une API a plusieurs consommateurs.
/v1/tasks /v2/tasks v1 -> ancien format v2 -> nouveau format
Idée clé plus une API est partagée, plus son évolution doit être contrôlée.
API Réelle

En production, une API vit avec logs, monitoring, gateway et déploiement

  • Une API Gateway peut centraliser l’entrée, la sécurité et certaines règles.
  • Le déploiement doit être reproductible pour éviter les surprises.
  • En pratique, l’architecture continue donc au-delà du code métier.
