DTO vs DAO

Deux noms qui se ressemblent, mais deux responsabilités très différentes dans une API Spring Boot.

API Spring Boot To-Do List Débutants
Problème concret

Une tâche n’a pas la même forme partout

  • Le client veut un JSON simple.
  • Le code métier veut manipuler une tâche.
  • La base veut lire et écrire des données.
  • On évite de tout mélanger dans une seule classe.
{ "title": "Faire les courses", "done": false }
Question Est-ce que ce JSON doit forcément ressembler à la table SQL ?
DTO

DTO = objet de transport

  • DTO signifie Data Transfer Object.
  • Il sert à transporter des données entre deux couches.
  • Il représente souvent le JSON reçu ou renvoyé.
  • Il ne contient pas la logique d’accès à la base.
public record TaskResponseDto( Long id, String title, boolean done ) {}
Image mentale Le DTO est le colis envoyé au client : propre, lisible, adapté au destinataire.
DAO

DAO = objet d’accès aux données

  • DAO signifie Data Access Object.
  • Il isole le code qui parle à la base.
  • Il cache les détails SQL ou JPA au reste de l’application.
  • Dans Spring, un Repository joue souvent ce rôle.
public interface TaskDao { List<TaskEntity> findAll(); TaskEntity save(TaskEntity task); void deleteById(Long id); }
Image mentale Le DAO est le guichet de la base de données.
Ne pas confondre

DTO parle au client, DAO parle à la base

  • DTO : format d’entrée ou de sortie de l’API.
  • DAO : opérations de lecture et d’écriture.
  • DTO = données qui circulent.
  • DAO = porte d’accès au stockage.
DTO DAO
Représente un message Représente un accès aux données
Proche du JSON Proche de la persistance
Visible côté API Interne côté serveur
Dans Spring Boot

Le Repository remplace souvent le DAO manuel

  • Avec Spring Data JPA, on écrit peu de code d’accès aux données.
  • L’interface Repository fournit déjà beaucoup de méthodes.
  • Le rôle reste le même : accéder à la base.
  • Le nom DAO est moins fréquent dans les projets Spring modernes.
@Repository public interface TaskRepository extends JpaRepository<TaskEntity, Long> { }
À retenir Dans un cours Spring, Repository et DAO désignent souvent la même zone de responsabilité.

Code court

DTO d’entrée et DTO de sortie

  • Le DTO d’entrée contient ce que le client a le droit d’envoyer.
  • Le DTO de sortie contient ce que l’API accepte de montrer.
  • On peut masquer certains champs internes.
  • On garde une API stable même si la base change.
public record CreateTaskDto( String title ) {} public record TaskResponseDto( Long id, String title, boolean done ) {}
Pourquoi séparer ?

Éviter de publier la structure interne

  • La base peut contenir des champs techniques.
  • Le client n’a pas besoin de tout voir.
  • Le JSON peut être plus simple que l’entité.
  • Le code devient plus facile à faire évoluer.
@Entity class TaskEntity { Long id; String title; boolean done; LocalDateTime createdAt; String internalStatus; }
Mini-réflexe Un champ interne en base n’est pas automatiquement un champ public dans l’API.
Erreur fréquente

Utiliser l’entité JPA comme DTO

  • C’est rapide au début.
  • Mais l’API devient dépendante de la base.
  • On risque d’exposer trop d’informations.
  • Les changements de table cassent les clients.

Comment choisir le bon objet ?

  • Objet envoyé par HTTP ? DTO.
  • Objet sauvegardé par JPA ? Entity.
  • Objet qui accède à la base ? DAO ou Repository.
  • Objet qui décide ? Service ou Use Case.

Résumé

La phrase à retenir

  • DTO : ce que l’application échange.
  • DAO : comment l’application récupère ou stocke.
  • Repository Spring : DAO moderne dans beaucoup de projets.
  • Séparer les rôles évite un code fragile.
Formule courte DTO = transport. DAO = accès aux données.
